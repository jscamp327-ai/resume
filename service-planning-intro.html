버전 혼란

참조 단절

정책 문서가 바뀌면 컴포넌트, 와이어프레임의 링크가 자동으로 끊깁니다.

PLY-001 → CMP-001 → WFR-001 ← 연결 추적 불가
💬

피드백 분산

검토 의견이 카카오톡·이메일·댓글에 흩어져 어떤 의견이 반영됐는지 파악이 어렵습니다.

카톡 / 이메일 / 댓글 / 구두 전달 ← 추적 불가
🚪

맥락 소실

담당자가 바뀌면 왜 이렇게 결정했는지 맥락이 사라져 처음부터 다시 논의합니다.

"이거 왜 이렇게 됐지?" ← 알 수 없음
3단계 구조

기획 의도가 문서가 되고,
문서가 이력이 됩니다

자연어 요구사항 → 구조화된 Markdown 문서 → Git 이력. 이 흐름이 서비스플래닝의 핵심입니다.

1

기획 의도

기획자의 자연어
# 기획자 말
"주문 내역 빈 화면이
필요해요"
→
2

문서 (Markdown)

Claude가 구조화
PLY-HOM-001 정책
CMP-HOM-001 컴포넌트
WFR-HOM-002 와이어프레임
→
3

이력 (Git)

자동 추적·보존
$ git log --oneline
a9f8725 fix: 빈화면 정책 추가
# 누가·언제·왜 남음
⏱
Git = 문서의 "타임머신" + 협업 도구
📄
Markdown = 누구나 읽을 수 있는 구조화된 기획서
🤖
Claude Code = 기획 의도를 문서로 바꿔주는 AI 파트너
비교

기존 방식 vs 이 방식

항목별로 어떻게 달라지는지 한눈에 비교합니다.

항목 ❌ 기존 PPT 방식 ✅ Claude Code + Git
버전 관리 파일명으로 구분 (혼란) Git 이력 자동 추적
문서 참조 수동 링크 (쉽게 끊김) ID 체계로 자동 연결
검토 의견 카카오톡 · 이메일 분산 Jira 티켓으로 추적
검색 파일 열어서 Ctrl+F 전체 키워드 검색 가능
와이어프레임 별도 툴 (Figma 등) HTML로 자동 렌더링
AI 활용 없음 Claude가 작성 · 검증 · 렌더링
담당자 교체 맥락 사라짐 커밋 이력에 이유 보존
실제 워크플로

기획 한 마디가
문서 세트로 완성됩니다

기획자가 요청하면 Claude가 단계별로 문서를 생성하고 Jira까지 연결합니다.

👤
기획자
"주문 내역 빈 화면이 필요해요. 일러스트 + CTA 버튼 형태로요."
Step 1
PLY

정책 문서 생성

빈 화면 노출 조건·정책 자동 작성

Step 2
CMP

컴포넌트 명세

일러스트 · CTA 버튼 명세 정리

Step 3
WFR

HTML 와이어프레임

브라우저에서 바로 확인 가능한 화면

Step 4
GIT

커밋 + Jira 연결

자동 커밋 + 티켓 Done 처리

발표 시 데모 포인트

이것만 보여주면
바로 납득합니다

실제로 터미널·브라우저·Jira를 열어서 4가지를 시연하면 가장 효과적입니다.

🗂

파일 트리

도메인 단위로 정책→컴포넌트→와이어프레임이 폴더 구조로 연결된 것을 보여줍니다.

tree domains/FO/home
📜

Git 로그

기획 변경 이력이 커밋 단위로 남아 누가·언제·왜 수정했는지 한눈에 보입니다.

git log --oneline
🖥

HTML 와이어프레임

별도 Figma 없이 브라우저에서 바로 와이어프레임을 열어 인터랙션까지 확인합니다.

open WFR-HOM-001.html
🎫

Jira 연동

커밋 메시지에 Jira 티켓 번호가 자동으로 연결되어 기획·개발 추적이 한 곳에 모입니다.

AFNBPB-1285 docs(home): ...
"
