STEP 01 · ORIGIN

하네스의 어원 — 말의 마구(馬具)

영어 harness는 원래 "말에 씌우는 마구"를 뜻해요. 말의 힘을 수레나 쟁기에 안전하게 연결해주는 장치죠. 이 이미지를 머릿속에 그려두면 앞으로 나올 모든 설명이 쉬워집니다.
🐎
말 (힘)
알맹이 · 엔진
+
🦯
마구 (하네스)
감싸서 연결
=
🛒
일하는 수레
쓸모 있는 결과

말 그 자체만으로는 수레를 끌 수 없어요. 말의 힘을 바깥 세계(수레·쟁기)와 연결해주는 "틀"이 필요하죠. 그게 바로 하네스입니다.

💡
핵심 직관 하나만 기억하세요

하네스 = "알맹이를 쓸 수 있게 해주는, 알맹이를 둘러싼 모든 것"

02
STEP 02 · CONCEPT

소프트웨어에서의 하네스

이 비유가 소프트웨어로 넘어오면서 "대상을 실행·제어·관찰할 수 있도록 감싸는 바깥쪽 틀"이라는 뜻이 됐어요. 분야마다 쓰임은 달라도 개념은 똑같습니다.

어디서 이 말을 쓸까?

🧪

Test Harness

테스트 대상 코드를 실행·입력 주입·결과 수집하는 러너. JUnit, pytest 같은 것들.

🤖

Agent Harness

LLM을 호출하고 도구·메모리·권한·루프를 연결해주는 실행 환경. Claude Code가 여기에 해당.

🔌

Wire Harness

임베디드 시스템에서 부품들을 연결하는 배선 다발. 가장 원형에 가까운 용례.

🛡️

Fuzz Harness

보안·퍼징에서 타겟 함수에 입력을 주입하는 최소 래퍼. 취약점 탐지용.

하네스는 이런 특징을 가져요

🎁
대상을 감싸되, 대상 자체는 아니다 — 엔진과 차체의 관계
🔀
입출력 통로를 제공한다 — 외부와의 연결 지점 (도구, 권한, I/O)
🔄
제어 루프를 돌린다 — 실행·관찰·개입이 가능하다
🧩
교체 가능하다 — 같은 대상도 다른 하네스로 돌리면 동작이 달라진다
🔑
이거 하나만 이해하면 됩니다

알맹이(엔진)와 감싸는 틀(하네스)은 서로 다른 존재. 같은 엔진도 하네스가 달라지면 완전히 다른 제품이 돼요.

03
STEP 03 · PLATFORM

Claude Code 하네스 — 이미 존재하는 플랫폼

Claude Code에서 말하는 "하네스"는 "Claude 모델을 감싼 CLI 실행 환경 전체"예요. 이건 Anthropic이 이미 만들어 놓은 것이고, 우리는 건드리지 않아요.

잠깐, CLI가 뭐예요?

용어 풀이

CLI = 명령줄 인터페이스

Command Line Interface의 줄임말. 마우스 클릭 대신 키보드로 명령어를 입력해서 컴퓨터와 대화하는 방식이에요.

우리가 흔히 보는 그 "검은 화면에 글자가 흐르는 창" — 바로 그게 CLI입니다.

CCommand LLine IInterface
Terminal — claude-code
$ claude ✨ Claude Code가 시작되었습니다.   > BO-PDM-001 정책 문서 만들어줘 

GUI vs CLI — 뭐가 다를까?

🖱️ GUI 우리에게 익숙한 것

Graphical User Interface. 버튼·아이콘·메뉴를 클릭·드래그해서 조작. 윈도우 탐색기, 크롬 브라우저 같은 것들.

⌨️ CLI 개발자가 쓰는 것

Command Line Interface. 키보드로 명령어를 타이핑해서 조작. 터미널, 파워셸, 그리고 Claude Code.

🤔
왜 Claude Code는 CLI일까?

개발자의 일상(코드 편집·git 커밋·테스트 실행)이 대부분 터미널에서 일어나기 때문이에요. Claude가 그 환경에 직접 들어와야 파일을 읽고, 명령을 실행하고, 작업을 이어갈 수 있거든요. 예쁜 버튼 UI가 아니라 작업 흐름에 자연스럽게 녹아드는 방식을 택한 거죠.

3.5
STEP 3.5 · CORE MECHANISM

컨텍스트 — 하네스가 엔진 앞에 펼쳐주는 '시야'

하네스가 엔진을 감싸고 있다는 건 알았어요. 그럼 하네스는 매 순간 엔진에게 무엇을 보여줄지를 어떻게 정할까요? 그 '보여주는 것'의 이름이 바로 컨텍스트(Context)입니다.

잠깐, 컨텍스트가 뭐예요?

용어 풀이

Context = 한 번에 보이는 정보 전체

Claude가 답을 만들기 직전, 모델의 '눈앞에 펼쳐진' 모든 글자를 뜻해요. 시스템 규칙, 지금까지의 대화, 읽어온 파일, 도구가 돌려준 결과 — 이게 다 한 덩어리로 묶여 모델에게 넘어갑니다.

쉽게 말하면 "이 시점 Claude의 시야(視野)"예요. 시야 안에 없으면, Claude는 그게 세상에 있는지조차 몰라요.

①시스템 프롬프트 ②대화 이력 ③읽어온 파일 ④도구 결과 ⑤주입된 메모리
context — snapshot
┌─ 지금 Claude 눈앞에 ─────┐ │ [시스템] 너는 기획 보조… │ │ [유저] BO-PDM-001 열어 │ │ [파일] policy.md (423줄)│ │ [메모리] MEMORY.md 요약 │ └──────────────────────────┘ > 이걸 전부 보고 답한다 
👁️
핵심 직관 하나만 더 기억해 주세요

시야 안에 있으면 안다, 시야 밖에 있으면 모른다. "Claude가 왜 내 파일을 몰라요?" — 99%는 그 파일이 컨텍스트에 안 들어가 있어서예요.

컨텍스트에는 뭐가 들어갈까?

📜

시스템 프롬프트

Claude에게 "너는 누구이고 어떤 규칙을 지켜야 하는지"를 알려주는 맨 앞의 지시문.

💬

대화 이력

이번 세션에서 지금까지 오간 모든 메시지. 길어지면 자동으로 요약(compact)됩니다.

📄

읽어온 파일

Read 도구로 불러온 파일의 내용. 읽지 않은 파일은 존재해도 '시야 밖'이에요.

🔧

도구 결과

Grep, Bash 등이 돌려준 검색 결과·실행 로그. 이것도 다음 턴에 컨텍스트로 남습니다.

🧠

주입된 메모리

CLAUDE.md · MEMORY.md처럼 세션 시작 시 자동으로 컨텍스트에 올라오는 프로젝트 기억.

🛠️

사용 가능한 도구 목록

지금 Claude가 쓸 수 있는 도구들의 이름과 설명. 이것도 시야에 포함돼요.

컨텍스트 윈도우 — '시야'에도 크기가 있어요

👁️ 컨텍스트 윈도우 (한 번에 담을 수 있는 양)
⚡ Claude 4.x 기준 · 최대치는 Opus 4.7

같은 Claude라도 버전별로 한 번에 볼 수 있는 토큰 수가 달라요. 시야가 넓으면 많이 담을 수 있지만, 그만큼 비용·지연시간이 늘어나는 트레이드오프가 있습니다.

모델 컨텍스트 윈도우 상대 크기 특성 언제 쓰면 좋을까
🦅 Opus 4.7 1,000,000 토큰 최대 5× 가장 넓은 시야 · 복잡한 추론 · 깊이 있는 분석 대규모 코드베이스 이해, 장문 기획 문서 교차 검토

⚡ Haiku 4.5 200,000 토큰 1× 가볍고 빠름 · 저렴 간단한 분류·추출, 대량 처리, 빠른 응답이 필요한 작업
💡
시야 ≠ 지능

Opus의 1M 토큰은 더 많이 담을 수 있다는 뜻이지, 자동으로 더 똑똑하다는 뜻은 아니에요. 짧은 작업이라면 Sonnet·Haiku가 충분히 빠르고 저렴합니다. 프로젝트 하네스의 역할 중 하나가 "작업에 맞는 모델을 고르는 것"이에요.

토큰 · 컨텍스트 윈도우 — 한 번 더 정리

🔤 토큰(Token) 최소 단위

Claude가 글을 세는 최소 단위예요. 글자나 단어보다 더 작은 조각이라 생각하시면 돼요. 한국어는 대략 한 글자 ≒ 1~2 토큰쯤. "1M 토큰"이라고 하면 대략 한국어 기준 책 2~3권 분량을 한 번에 담을 수 있다는 뜻이에요.

👁️ 컨텍스트 윈도우 시야의 크기

모델이 한 번에 볼 수 있는 최대 토큰 수. 창문(window) 너머로 보이는 풍경의 크기라고 생각하면 돼요. 창문이 크면 더 많이 보이지만, 그만큼 더 많이 읽고 생각해야 해서 비용과 시간이 늘어나요.

그래서, 하네스와 컨텍스트는 무슨 사이?

하네스 (틀)

무엇을 언제 넣을지 정한다

  • 파일을 언제 Read할지 결정
  • 메모리를 세션 시작에 주입
  • 도구 결과를 다음 턴에 전달
  • 길어진 대화를 자동 요약
컨텍스트 (내용물)

그 결과로 실제 모델 앞에 놓인 것

  • 매 턴마다 새로 구성됨
  • 윈도우 크기에 제한됨
  • 여기에 없으면 Claude는 모름
  • 품질은 전적으로 하네스에 달림
" 같은 엔진, 같은 하네스라도
컨텍스트에 뭐가 들어가느냐에 따라
결과는 완전히 달라집니다.
✅
프로젝트 관점으로 한 줄만

각 프로젝트의 CLAUDE.md·MEMORY.md·스킬·에이전트는 결국 "이 프로젝트에서 Claude의 시야에 뭐가 들어갈지"를 설계하는 장치들이에요.

프로젝트 하네스 — 각 프로젝트가 조립하는 레이어

Claude Code 하네스가 열어놓은 확장 슬롯(skills · agents · settings · MCP · memory)에, 각 프로젝트는 자기 도메인에 맞는 내용물을 채워넣습니다. 그렇게 완성된 것이 바로 "프로젝트 하네스"예요.
👁️
Chapter 3.5에서 이어서

프로젝트 하네스가 하는 일 = "이 프로젝트에서 Claude의 시야(컨텍스트)에 뭐가 들어갈지"를 설계하는 것. 아래의 6개 슬롯은 모두 그 시야를 채우고 · 비우고 · 기억시키는 장치예요.

이 비교에서 얻을 수 있는 3가지 인사이트

INSIGHT 01
하네스는 "있고/없고"가 아니다

harness-plugins 같은 메타 레벨부터 welfare-mall·service-planning 같은 풀스택 도메인 하네스까지, 복잡도 스펙트럼이 존재한다. 프로젝트 성격에 맞게 투자 수준을 선택한다.

INSIGHT 02
목적이 하네스의 모양을 결정한다

management는 개발자(fe-developer·qa-tester), 두 기획 프로젝트는 기획자(planner·publisher·uiux-expert). 에이전트 이름만 봐도 프로젝트의 정체성이 드러난다.

INSIGHT 03
같은 구조가 도메인별로 복제된다

welfare-mall(메가존플레이 BO)과 service-planning(스마트면세 F&B)은 동일한 6단계 파이프라인을 쓰지만, 도메인 특성에 맞춰 에이전트·스킬·스키마를 각기 다르게 조립한다.

🔭
특별한 케이스 — harness-plugins

D:\VSCODE\harness-plugins는 다른 프로젝트와 완전히 다른 층위에 있어요. 이건 "하네스 구성요소를 플러그인으로 만들어 배포하는 저장소", 즉 하네스를 만드는 하네스입니다. .claude-plugin/marketplace.json으로 여러 프로젝트가 공통으로 쓸 수 있는 하네스 부품을 관리하죠.

★ Key Takeaway ★

한 문장으로 압축하면

하네스는 "알맹이를 감싸는 실행 환경"이고,
컨텍스트는 그 하네스가 매 순간 엔진 앞에 밀어 넣는 '시야'예요.
각 프로젝트는 Claude Code 플랫폼 위에
자기만의 하네스를 구성해서 — 즉, 자기 시야를 설계해서 —
도메인에 맞는 산출물을 생산한다.

플랫폼은 이미 존재하고, 각 프로젝트가 할 일은
도메인 특화 레이어를 조립해서
Claude의 시야(컨텍스트)를 설계하는 것입니다.
